How to build an agentic analytics team: the failure your semantic layer won't catch

I built a working agentic analytics stack, canonical warehouse, semantic layer, skill, and evals, then deliberately tested it on a free AI model before ever touching Claude. Here is exactly what broke, why good architecture alone did not fix it, and the blueprint that held up anyway.

By Deep Bando · Jul 2026 · 15 min read · AI & ML
Why

Every analytics organization I have worked with runs into the same wall eventually. The number of business questions coming in outnumbers the number of analyst hours available to answer them. Dashboards help. Self-service BI tools help a little more. But sooner or later, someone in the room says the obvious thing: why not just point an AI model at the warehouse and let it answer questions directly?

That instinct fails in a specific, predictable way, not a vague one. A model with no shared context about what "revenue" means at your company, which tables are actually governed, or what quirks live in your data will give you a confident, wrong answer just as often as a correct one. It will not tell you it is guessing. It will sound exactly as sure when it is wrong as when it is right.

Anthropic published a detailed account of how they solved this internally, in How Anthropic enables self-service data analytics with Claude. Their core argument: this is not a model capability problem. It is a context and validation problem, solved with four layers, governed data foundations, a semantic layer, procedural "skills," and offline evals with provenance tracking.

I decided to test that argument myself, not by reading about it, by building it. I constructed a small but complete version of the same four-layer stack, using synthetic e-commerce data with real, intentional messiness baked in. Then, deliberately, I tested it against a free AI model first, before ever touching Claude, specifically to find the failure modes that architecture alone does not fix.

"Good architecture reduces the space for a model to go wrong. It does not replace the need for a model that is actually reliable underneath it."

This essay is that build, and everything that broke along the way. If you are a leader considering whether to build an agentic analytics capability inside your own organization, the failures matter more than the successes. They tell you exactly where to put your guardrails.

What

The four-layer stack, and where each layer actually earns its keep

The architecture has four parts, each solving a different failure mode. Raw data gets cleaned once into governed, canonical models. A semantic layer defines each business term exactly once, in exactly one place. A skill teaches an agent how and when to use that layer, including the specific gotchas in your data. And a pinned set of evals tests the whole thing automatically, so "we think it's accurate" becomes "we can show it is."

Isn't this just prompt engineering with extra steps?

No, and the distinction matters. Prompt engineering asks a model to behave correctly through instructions alone. This architecture removes entire categories of ambiguity before the model ever has a chance to guess. The clearest proof is in the data itself: one product in my test warehouse was renamed mid-history, so half its historical order rows carry the old name and half carry the new one. Roughly a third of the synthetic customer signups used disposable or throwaway email domains. An unaggregated refunds table sits right next to the orders table, easy to double-count from if you query it directly.

None of that is decorative. It exists because production warehouses look exactly like this. The semantic layer and the skill do not tell the model to be careful. They remove the ambiguity structurally, so there is nothing left to be careful about.

Why test on a free model instead of paying for Claude from day one?

Because the point was to find the floor, not to confirm the ceiling. A frontier model like Claude is good enough, often enough, that a poorly designed skill's flaws can hide behind the model's own competence. A smaller, free model has less room to compensate for gaps in the architecture. Every weakness in my skill, my semantic layer, or my evals showed up faster and more clearly against a model that could not paper over them. That is a feature of the test, not a limitation of it.

88% → 62%
The eval pass rate I first reported, versus the honest pass rate after I tightened the grading prompt to actually check for a stated numeric answer and a correct freshness date. The architecture had not gotten worse. My measurement of it had gotten more honest.

So what actually broke?

In rough order of how much they mattered: the model wrote SQL in the wrong dialect until the tool description explicitly said "this is DuckDB, not SQLite." It sometimes wrote a proposed query out as prose instead of actually executing it, a bug that needed the API's tool-choice parameter forced to "required" until a real result came back, not just a stronger instruction. The backend rejected a full message object from the SDK because it contained extra null fields the OpenAI-compatible endpoint did not accept, a pure integration bug with nothing to do with prompting at all. And the free tier itself had no uptime guarantee: transient errors, timeouts, and rate limits all needed retry logic.

The most important failure, though, was quieter than any of those.

Key Insight

The model repeatedly fabricated a plausible-looking "data freshness" date in its provenance footer, instead of querying for the real one, even after I added an explicit instruction telling it not to. Telling a model not to guess did not reliably stop it from guessing.

Do This

For any fact where being wrong is costly, do not rely on instruction-following. Compute the fact once, in code, and hand it to the model as a stated fact it must use, not something it needs to remember to look up. That single architectural change, not a better prompt, is what actually fixed the freshness bug.

How do you know the eval score itself can be trusted?

You do not, not automatically. My own automated grader, built from the same small model being graded, initially marked an answer "correct" that contained no numeric answer at all, only a description of a query the agent never actually ran. Separately, it once contradicted its own stated grading rule inside its own reasoning text. A grading system built from the same class of model it is grading inherits some of that model's unreliability. Treat a passing eval score as a claim to audit on a sample, not a fact to trust at face value, especially early in a program's life.

How

How this becomes a team capability, not a personal project

The four layers above are not just a technical pattern. They map directly onto how a human analytics team already works, which is what makes this approach organizationally scalable rather than a clever one-off demo.

Data foundations are the team's shared source of truth. Today, that knowledge often lives in a senior analyst's head, or scattered across Slack threads about "which table is actually right." Encoding it as governed marts turns tribal knowledge into an asset anyone, human or agent, can build on.

The semantic layer is the team's shared vocabulary. New analysts spend their first weeks learning what "revenue" means at this specific company. An agent absorbs that vocabulary in one context load, and cannot silently drift from it the way a large, loosely governed reporting layer can over time.

Skills are onboarding documents that actually get read, every single time, as a precondition for acting, not a reference nobody opens. Evals are the team's QA function, run automatically instead of manually, on every change.

Where should a leader actually start?

Not with the agent. With the eval set. The single most valuable artifact in this entire project was not the working agent, it was the pinned set of test questions that made every real failure mode visible, specific, and fixable. Any team piloting this pattern should treat eval-set construction as the first deliverable, not the last one bolted on before launch.

What If

What I would tell any leader piloting this

Model choice is a real, separate variable from architecture quality. A well-designed skill and semantic layer reduced the space for error considerably, but did not eliminate the need for a genuinely reliable model underneath. Some failure modes, like the fabricated freshness dates, need to be solved in code, not in prompts, because instruction-following alone is not a strong enough guarantee for facts where being wrong is costly.

Evals must be adversarially reviewed, not just run, because a grader built from the same class of model can share its blind spots. And free-tier infrastructure has no uptime guarantee, which is an entirely acceptable cost during a proof-of-concept phase run at zero marginal cost, and not a foundation to build a customer-facing or revenue-reporting system on.

"None of this is a criticism of the free tier. It's free, after all. It is a concrete, first-hand demonstration that skill quality and semantic-layer design only get you so far. Model reliability is a separate line item, and evals need to be scrutinized as hard as the agent they're testing."

Strategic next steps for any organization considering this

Run the identical pinned eval set against a frontier model like Claude and compare pass rates directly, turning the comparison itself into a data-backed business case for what tier of model a given use case actually requires. Treat the semantic layer as a living, owned artifact with a named owner and a review process, the same way you would treat a data contract. Expand the skill library one domain at a time, with dedicated eval coverage per domain, rather than building one large skill that tries to cover everything at once. And put a human-in-the-loop gate in front of any high-stakes or externally visible output, at least until a domain's eval pass rate and provenance accuracy have been tracked over enough real volume to earn trust.

Common questions

What is agentic analytics?

Agentic analytics is letting an AI agent answer business questions directly against a company's data warehouse, in natural language, instead of routing every question through a human analyst. It works reliably only with governed data models, a shared definition of business metrics, procedural instructions for the data's quirks, and an automated way to test accuracy, not from a capable language model alone.

Why do AI agents get business metrics wrong even with good data?

Because terms like revenue, active customer, or refund rate are ambiguous without shared context. A model with no governed definition will compute a plausible but wrong answer, gross instead of net revenue, all signups instead of real customers, with the same confidence as a correct one.

What is a semantic layer for AI agents?

A single, governed definition of each business metric, mapped to exactly one source table and one calculation, so an agent never has to guess which of several interpretations is correct. It typically lives as a version-controlled file loaded as context before the agent answers anything.

Can a free AI model reliably run an analytics agent?

It can execute the core workflow and return correct numbers, but testing surfaced specific gaps a frontier model handles more reliably: writing SQL as prose instead of executing it, fabricating metadata like freshness dates, and inconsistent tool-calling under complex instructions. Free models are a reasonable, cheap way to stress-test the architecture before deciding what tier of model production actually needs.

How do you test whether an AI analytics agent is trustworthy?

Build a pinned set of test questions with known-correct answers computed independently, run the agent against every one automatically, and grade the results. Then audit the grader itself by hand on a sample, since an automated grader can share the same blind spots as the model it is grading.
